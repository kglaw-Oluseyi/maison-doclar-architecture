**Deliverable 06B of the Strategic Architecture Program**
Status: Draft for principal review · Date: 11 July 2026
Governing rule (in force for every assignment): *Repository classification is justified exclusively by the knowledge it realizes, preserves, donates, or no longer uniquely contributes. Age, code quality, and implementation language have no standing.* Categories: **Living system · Signature lineage · Platform donor · Knowledge donor · Legacy execution system · Experimental system · Reference implementation · Deprecated · Read-only.** There is no merge category. Standing safety rule: no repository is archived, made read-only, deleted, or modified without your explicit per-repository instruction. This document assigns classifications and trajectories; it executes nothing.

## 3. Classification mechanics (definitions, so status is never ambiguous)

*Living system*: receives features and fixes; on-call surface. *Signature lineage*: bound to Signature clients' needs; divergence-registered. *Platform donor*: named artifacts scheduled for knowledge extraction into baseline/Platform design. *Knowledge donor*: extraction is the repo's remaining purpose. *Legacy execution system*: runs, maintained for security/stability only, no new capabilities. *Reference implementation*: kept readable as the worked example of a preserved pattern. *Experimental*: optionality preserved deliberately, no current investment. *Deprecated*: successor identified; no investment; removal from operation planned. *Read-only*: branch-protected, no writes, permanently consultable — applied only by your instruction, never by default; history is never rewritten; nothing is deleted.
