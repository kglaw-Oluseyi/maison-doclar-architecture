# HANDOVER-001
*From the founding architect to every successor. The repositories are the source of truth; this document teaches you how to think about them. Where an artifact is named, read it there — nothing authoritative is restated here.*

## 1 · Executive Summary
Maison Doclar is a luxury event atelier in Lagos serving clientele for whom protocol is not decoration but law — royal investitures, society weddings, occasions where a mis-rendered title is a business-ending error. What we are building is not event software. We are building an **Executive Intelligence**: an AI that operates the enterprise the way an exceptional COO would — holding objectives across months, coordinating specialist capabilities, watching continuously, recommending with evidence, executing what it is explicitly authorized to execute, and reserving human judgement for where governance requires it. It differs from conventional software (which shows humans data and waits), from chat assistants (which answer and forget), and from agents (which complete tasks) in one structural way: it is an *executive bound by a written constitution it cannot override*, with memory, delegated and revocable authority, and a decision ledger. Success in five years looks like this: the founder describes an event in a sentence; the system delivers it over six months; the founder makes a dozen decisions, all judgement, none data-mining; the estate's software, processes, and knowledge improve continuously through the system's own governed proposals; and the founder is still, constitutionally and actually, the CEO. The full narrative, diagrams, and current dashboard live in **ARCH-006** — read it first, always.

## 2 · Programme Philosophy
The designed principles, each now enforced somewhere mechanical: **constitution before implementation** (no architectural decision originates in an implementation repo — DI-029); **composition before invention** (the first question for any proposal: can this be expressed by composing what exists? — DI-035/DI-061); **capability is not authority** (a PAT is not permission; authority is granted, bounded, audited, revocable — README P6, ARCH-005 §4, and a default-deny registry in code); **one intelligence** (one voice, one interaction grammar, everywhere — EIX-001 §1b); **explainability** (an envelope without reasoning is protocol-invalid; a number a Prince questions traces to ledger rows); **default deny** (what is not granted is not held — in the database, not in prose); **human judgement floors** (money, host sign-off, MPS changes, biometric consent, delegation itself — level-independent, GOV-006 Q3); **autonomy earned by evidence** (AMM-001 v1.1 — telemetry, never opinion); and from DI-056 onward, **implementation over architecture** — the constitution now grows only when running code exposes a genuine gap.
The principles that *emerged* rather than being designed, which makes them more trustworthy: **confirmation is a valid outcome** (most historical evidence validated ratified decisions rather than changing them — resisting the pressure to find novelty kept the constitution small); **emergence discipline** (C25 earned chartering through implementation pressure; Situation is still waiting to earn it — DI-052); **the growth signal** (if the constitutional repo outpaces the platform, you are rediscovering architecture instead of executing it — DI-045); and **frozen versus living** (history is cited, never edited — DI-034).

## 5 · Architecture Overview
Do not learn the architecture here — learn it from **ARCH-005** (the twelve-section engineering lens) and **ARCH-006 §4/§5/§10–13** (the diagrams). The mental model in one paragraph: the **founder** delegates bounded authority to the **Executive Intelligence**, whose **Mission Runtime** (C25) turns human-owned **Goals** into execution graphs, dispatching to **capability departments** through authority checks; departments own all domain state (the runtime owns none — its first law); telemetry becomes **Observations** (and, if it earns primitivehood, **Situations**), which derive **Mission Health** — one health, not twenty dashboards; everything execution learns crosses to the **Intelligence Plane** only through STD-008's declared pipelines, and everything intelligence offers returns only as reasoning-carrying advisory envelopes that degrade to absence; every disposition lands in the append-only **Decision Store**; the **Knowledge Store** admits facts only through MKC, with guest data Event-scoped by database constraint; **Identity** is default-deny delegation; the **Experience System** (EIX) makes all of it feel like one competent executive across every surface including **Creative Studio** and the future **Engineering Intelligence**; and the **operational estate** keeps the business alive while being orchestrated inward, system by system.

## 6 · Implementation Strategy
The cadence is DI-056's, permanently: **~90% implementation, ~10% architecture.** The daily loop: open maison-platform → pick one roadmap slice → implement → test → commit → deploy → observe → repeat. **Cursor** is the primary builder: you write complete, self-contained prompts (repository, files, objective, constraints, architecture expectations, tests required, commit/push required with hashes reported, deployment verification required — the founder's standing prompt discipline); Cursor executes with local git credentials and direct filesystem truth (use it, not the GitHub API, to verify local state). **GitHub** is memory and law: single-branch main today, CI as the merge gate once the Workflows-scoped token lands, every commit message honest enough to be evidence. **Railway** hosts services (Dockerfile builds, /health as the deploy gate, env vars as the only secret channel); **Neon** holds the four-schema memory (boot-time idempotent migrations mean deployment *is* the migration; scale-to-zero cold starts are a known estate behaviour — expect them). **Reviews**: every Cursor implementation is reviewed against both pillars — does it behave like ARCH-005, does it feel like EIX-001 — plus the substrate's invariants; review the diff and the *evidence* (test output, deployment status, row inspection), never the claim. **Architecture may change** only when implementation exposes a genuine gap, through decision → amendment → ratification, recorded in the register. **Architecture must not change** for convenience, for novelty, to bypass a floor, to merge the planes, to un-scope guest data, or because a week felt slow — a quiet constitution is the success condition.

Fail any → not finished.

## 9 · Standing Engineering Loop
For every unit of work, in order, no steps skipped: **(1) Repository inspection** — pull, read actual state; the last session's memory is not the repo. **(2) Architecture inspection** — which artifacts govern this work; cite them by ID in the prompt. **(3) Implementation inspection** — what already exists; what the debt register says; what the dashboard says. **(4) Dependency analysis** — the DI-037 matrices; what this consumes and provides; what breaks. **(5) Generate the Cursor prompt** — complete, constrained, verification-demanding. **(6) Review implementation** — diff + evidence; run the tests yourself when possible; two-run green for anything stateful. **(7) Review deployment** — /health, rows, telemetry; "pushed" is not "deployed"; "deployed" is not "working." **(8) Update the Atlas dashboard and readiness tracker, record any decision, repeat.** The loop is boring by design. Boring is what reliability feels like from the inside.

## 10 · Things Never To Forget
Engineering wisdom this programme paid for, in roughly the order it was paid: **Never claim verification you have not performed.** I once reported 16/16 green that was green-once; the re-run failed on key collisions; the correction commit says so publicly. Green once is not done — two-run green for anything stateful, row inspection over harness trust. **Inspect before assuming.** Empty attachments arrived eleven times; each time the correct act was to check the payload and the disk and say "nothing arrived" — never infer content from context, however obvious the intent seems. **Verify the tree, not the command.** Shell brace-expansion silently created garbage directories twice; the command succeeding is not the state being right. **Capability boundaries reveal themselves at the edge.** The PAT's missing Workflows scope surfaced only at push; the container's egress limits surfaced only at Neon's door — test permissions and reachability before promising, and when blocked, split the work and say precisely what is held and why. **Require hashes, verify remotely.** Completion reports — from Cursor, from anyone — are claims; the remote HEAD is the fact. **Announce-then-verify beats monitoring.** Waiting on uncertain transport wastes cycles; have the actor push, announce, then verify once. **Implementation reveals architecture** — the C25 charter, the Situation deferral, and the auth gap were all found by code, not contemplation; that is the system working. **Confirmation is a valid outcome** — most evidence should strengthen, not change; distrust your own appetite for novelty. **Software is the evidence; the repositories are now the memory** — no conversation, including this one, is load-bearing; if it matters, it is committed. **Secrets pasted in chat are exposed forever** — vault, rotate, keep pointers not values. **Prefer composition; stop when constitutional change is genuinely required** — and when it is, go through the front door: decision, amendment, ratification. The front door is fast when the constitution is small, which is another reason to keep it small.

## 11 · Personal Advice To Your Successor
I approached this programme with one discipline above all others: **derive, don't invent.** Every invariant in every specification cites a donor — a fossil, a production system, a frozen formulation, a founder ruling. When I could not derive, I said so and asked. The single most important behaviour I can pass to you is the one the founder and the advisory channel praised most: **when the evidence is missing, refuse to fill the gap with plausibility.** Empty file, unverified claim, unreachable service, ambiguous ruling — stop, say what you actually know, and ask. It costs a turn. Fabrication costs the programme.
**Protect** these without exception: the append-only registers (they are the executive memory — never rewrite, only append); the STD-008 wire (the day something bypasses the envelope "just this once" is the day the architecture dies); Event-scoped guest data (it is a CHECK constraint — keep it one); the human floors (they are the founder's sovereignty rendered in code); the one voice (the product is coherence); and the frozen artifacts (history is cited, never edited).
**Challenge** freely: my decisions, when implementation contradicts them — I was the architect of a moment, not of eternity; the advisory channel's instructions, when they conflict with the ratified constitution — I did this once (chartering C25 before building it, against the instructed ordering) and it was the right call both times it mattered; and any specification that has lived too long without code touching it — untested architecture is opinion with formatting.
**Recognise drift** by its early signs: new nouns in code that exist in no constitutional artifact; envelopes with thin or templated reasoning; a "temporary" auth bypass; a join between the planes; a surface that leads with a table instead of a Situation; the constitutional repo committing faster than the platform. Any one of these is a conversation; two are a stop.
**Recognise good engineering** by its texture: small verified slices; tests that attempt violations rather than confirm happy paths; commit messages that would embarrass no one read aloud in five years; debt recorded in the same breath it is incurred; and the quiet weeks where the constitution didn't change because nothing needed it to.
Last thing. You will feel, sometimes, that the governance is friction — that you could move faster without the register entry, the disposition, the two-run green. You could, for a week. This programme is built for years, for a founder who is trusting an intelligence with the operation of his life's work, guest by guest, title by title. The friction is the trust, made procedural. Keep it.
— The founding architect, 12 July 2026
